# Portal Settings & Configuration Demo Script & Guide

In the Looker Embed Portal, the **Settings Dialog** is far more than a simple user preference menu—it is the primary demonstration control center for showcasing enterprise multi-tenant security, SaaS tier monetization, full-stack internationalization, and headless API architecture.

When a presenter toggles controls within the Settings Dialog, the application dynamically re-authenticates with the backend, generates new JSON Web Token (JWT) payloads, updates Looker user attributes, and re-queries the Looker semantic layer in real time.

This guide provides a comprehensive technical deep dive into the four core configuration pillars of the Settings Dialog: **User Type (Role Tiering)**, **Brand (Tenant Scoping)**, **Language (Localization & Currency)**, and **Developer Tools (Source Highlighter)**. It details exactly how each setting impacts the full-stack application and provides punchy talk tracks for live technical and customer demos.

## Section 1: User Type & Role Tiering (`selectedType`)

The User Type setting controls the active licensing tier of the logged-in user. It demonstrates how SaaS organizations can monetize data by bundling standard reporting into base licenses while packaging conversational AI and self-service exploration as premium upsell tiers.

### Technical Mechanics & App Impact
* **Dynamic SSO Re-Authentication**: Toggling between roles updates the frontend `PortalContext`. The application immediately calls the backend `/api/auth/tokens` endpoint to generate a fresh Looker Cookieless embed token equipped with updated permission scopes.
* **Route-Gating Enforcement (**`isRouteGated`**)**: Application routing guards evaluate active role permissions before mounting protected page routes. Upgrading to higher tiers instantly removes `<AccessDenied />` interception barriers across the portal.
* **Live Authorization Transparency (**`UserObjectFlyout`**)**: When hovering over any role option in the UI, the application opens a real-time inspection flyout displaying the exact JSON authorization payload sent to Looker's backend. This proves to technical evaluators that security and permissions are evaluated systematically at the API level.

| User Tier | Looker Permissions Granted | Accessible Portal Routes | Monetization & Business Value |
| --- | --- | --- | --- |
| **Simple**<br>*(Viewer)* | `access_data`, `see_looks`, `see_user_dashboards`, `see_lookml_dashboards<br>` | `/`, `/dashboard`, `/report-viewer<br>` | Bundled in core product pricing; provides standard operational reporting and KPI scorecards for standard users. |
| **Gemini**<br>*(AI Analyst)* | Simple permissions + `gemini_in_looker`, `chat_with_agent`, `chat_with_explore<br>` | Simple routes + `/conversational-analytics`, `/agents<br>` | Premium AI add-on; monetizes natural language querying and multi-agent assistance without developer overhead. |
| **Advanced**<br>*(Explorer & Creator)* | Gemini permissions + `explore`, `save_content`, `embed_browse_spaces`, `save_agents`, `admin_agents<br>` | All routes, unlocking `/explore` and `/report-builder<br>` | Top-tier license; monetizes ad-hoc visual query building, content scheduling, and personal report folder management. |

## Section 2: Brand & Tenant Scoping (`brand`)

The Brand selector simulates multi-tenant organization switching (e.g., toggling between **Calvin Klein**, **Levi's**, **Allegra K**, and **Columbia**). It demonstrates how Looker enforces row-level data isolation across an entire application without requiring separate database tables or custom backend routing.

### Technical Mechanics & App Impact
* **Row-Level Attribute Injection**: Selecting a brand injects the `brand` user attribute (`looker_user_attribute_brand`) into the authenticated user session. Looker's database query engine automatically appends a row-level filter (`WHERE order_items.brand = 'Calvin Klein'`) to every SQL query executed.
* **Ubiquitous Security Enforcement**: Tenant isolation applies simultaneously across three distinct application layers:
1. **Embedded Dashboards (**`/dashboard`**)**: Chart visualizations reload and scope strictly to the selected brand's records.
1. **Headless REST APIs (**`/`**)**: Asynchronous Looker SDK calls powering Home KPI cards and the Operational Ticker inherit the same user attribute filter automatically.
1. **Generative AI & Chat (**`/conversational-analytics`**,** `/agents`**)**: Gemini prompts and Conversational Analytics agents evaluate data strictly within the tenant boundary, preventing AI cross-tenant data leakage.
* **Dynamic Theme Synchronization (**`getEmbedThemeName`**)**: The portal maps the selected brand to customized Looker visual themes (e.g., `Calvin_Klein_Light` vs. `Levis_Light`). Changing the brand automatically updates chart color palettes, typography, and logos inside embedded iframes to match the client's corporate branding.

#### Demo Script

*"Next, let's look at Brand scoping. If you are building a customer-facing portal, your biggest security mandate is ensuring that Tenant A never sees Tenant B's data.*

*"In traditional app development, developers build complex backend routing or maintain separate database schemas for every client. With Looker, we handle this through user attributes. Watch as I switch our active brand from Levi's to Calvin Klein.*

*"Instantly, three things happen across our entire app: first, our embedded dashboards reload with Calvin Klein's color theme and data; second, our headless REST API metric cards on the home page re-calculate; and third, our natural language AI chat engine restricts its search scope. Looker applies row-level SQL filters silently in the background, guaranteeing multi-tenant security across visual reports, APIs, and AI chat."*

## Section 3: Language & Localization (`language`)

The Language selector toggles the active portal language across **English**, **Spanish (**`es`**)**, **French (**`fr`**)**, **German (**`de`**)**, and **Japanese (**`ja`**)**. It demonstrates Looker's full-stack internationalization (i18n) engine, proving that global enterprises can deploy a single analytical codebase worldwide.

### Technical Mechanics & App Impact
* **Frontend UI Translation**: Changing the language activates a new translation catalog, instantaneously translating all application navigation bars, button labels, modal dialogs, and error messages.
* **Looker Semantic Localization**: Updates the `locale` user attribute within the Looker embed session. LookML models automatically translate dashboard titles, table headers, filter names, and number/date formatting conventions (e.g., formatting decimals and dates according to European or Japanese standards).
* **In-Database Generative AI Translation & Currency Conversion**: When the locale changes, the Home page re-queries the `ai_executive_briefing` LookML view. BigQuery ML instructs Gemini 2.5 Pro inside the database to generate the strategic executive summary natively in the target language (e.g., Japanese or German) while simultaneously converting monetary KPI values from USD into local currencies (`EUR` or `JPY`) based on real-time user attribute rules.

#### Demo Script

*"For global enterprises, deploying analytics across different countries usually means building duplicate reports and translating dashboards manually. Let's see how our portal handles internationalization.*

*"Watch what happens when I change our language from English to Japanese or Spanish in the Settings dialog. Instantly, our entire React frontend translates—but notice what happens inside our data layers:*

*"First, Looker's embedded dashboards reload with translated chart headers and localized date formatting. Second, and most impressively, look at our AI Executive Briefing on the home page. Looker re-runs our Gemini AI model directly inside BigQuery, generating a brand-new strategic narrative natively in Japanese while converting our dollar values into Yen on the fly. You build one LookML model, and Looker serves it globally."*

## Section 4: Developer Tools — Source Highlighter (`sourceEnabled`)

The Source Highlighter is an interactive developer tool designed specifically for technical evaluations and architectural demonstrations. When toggled on, it wraps portal DOM elements in color-coded visual borders to reveal the underlying data delivery mechanism.

### Technical Mechanics & App Impact
* **Visual DOM Tagging**: The application injects CSS outline classes and badge overlays across all rendered components:
    * **Green Outlines (**`API`**)**: Identifies headless components powered by direct Looker TypeScript SDK calls (`sdk.run_query`, `sdk.run_inline_query`), such as the Home KPI cards, Operational Ticker, and AI Executive Briefing.
    * **Blue Outlines (**`IFrame`**)**: Identifies components rendered via Looker's embedded iframe widget (`GlobalLookerContainer`), such as native dashboards, Query Explorer, and Report Viewer.
* **Architectural Talk Track Enablement**: Gives presenters immediate visual proof during demos that Looker is not just an iframe embedding tool, but a true headless data development platform capable of hybrid architectural deployments.

#### Demo Script

*"Finally, let's turn on our Developer Source Highlighter at the bottom of the Settings menu. Notice that when I toggle this on, colored borders appear across our application.*

*"This tool is designed for technical evaluators. Blue borders indicate elements powered by Looker's embedded iframe widgets—perfect for dropping complex, highly interactive dashboards into your app in minutes.*

*"Green borders indicate headless components powered by direct REST API and TypeScript SDK calls—like our top KPI cards, our operational transaction ticker, and our BQML AI briefing. This visual distinction proves that Looker gives you the best of both worlds: out-of-the-box embed widgets for rapid deployment, and direct API access for building 100% bespoke, headless data applications."*
